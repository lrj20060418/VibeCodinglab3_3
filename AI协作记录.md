## 第一轮：嵌入高德地图，点击获取经纬度（并对齐评分补充地名）

### 需求

- 页面嵌入高德地图。
- 点击地图获取经纬度，**有可见反馈**（页面展示或控制台输出）。
- 为对齐评分「地名/经纬度」，补充逆地理编码显示地名。

### 实现要点

- 使用 `loader.js` + `AMapLoader.load` 加载 JS API 2.0。
- 点击事件读取 `e.lnglat`。
- `AMap.Geocoder` 逆地理编码：经纬度 → `formattedAddress`。
- UI：在页面面板中显示经纬度/地名，并在控制台输出日志。

### 主要文件

- `lab3/index.html`
- `lab3/js/app.js`
- `lab3/js/config.js`
- `lab3/css/style.css`

## 第二轮：接入天气 API（改为使用高德天气 Web 服务）

### 需求

- 点击地图后自动显示天气。
- 天气 API 使用 **高德**（非和风）。

### 实现要点

- 逆地理编码结果中读取 `adcode`。
- 调用高德天气接口：
  - `GET https://restapi.amap.com/v3/weather/weatherInfo?city=ADCODE&key=AMAP_WEBSERVICE_KEY&extensions=base`
- 页面展示字段：
  - 状态、温度、风、湿度、观测时间

### 配置要点

- 高德通常需要 **两类 Key**：
  - Web 端（JS API）Key：地图加载（含安全密钥 `securityJsCode`）
  - Web 服务 Key：天气查询（`AMAP_WEBSERVICE_KEY`）

### 主要文件

- `lab3/js/app.js`
- `lab3/js/config.js`
- `lab3/index.html`（天气展示面板）

## 第三轮：接入大模型 API（带天气上下文的 AI 对话，右侧栏）

### 需求

- 接入大模型 API，实现 AI 对话。
- 每次对话要带上「当前选点 + 当前天气」上下文。
- 专门安排一个 **侧栏** 用于 AI 对话输入与展示。

### 实现要点

- 侧栏 UI：
  - 消息列表（role log）
  - 输入框（textarea）
  - 发送/清空按钮
- 对话上下文注入：
  - 在发送请求时，把「地名/经纬度/adcode」与「天气（状态/温度/湿度/风/观测时间）」拼为上下文文本
  - 以 `system` 消息形式注入，确保模型稳定使用上下文
- 兼容 OpenAI 风格 `chat/completions`：
  - `POST LLM_BASE_URL`
  - `Authorization: Bearer LLM_API_KEY`
  - `model: LLM_MODEL`

### 主要文件

- `lab3/index.html`（右侧栏）
- `lab3/css/style.css`（两栏布局与聊天样式）
- `lab3/js/app.js`（聊天逻辑、上下文构建、调用大模型）
- `lab3/js/config.js`（新增 `LLM_*` 占位符）

## LLM 输出 Markdown 渲染（类似 md→pdf 的字号/粗细层级）

### 需求

- 将 LLM 返回的 Markdown 解析渲染，呈现不同字号、粗细、列表层级。

### 实现要点

- 轻量、安全的 Markdown 渲染：
  - 先进行 HTML 转义，避免注入
  - 支持：标题（#~###）、段落、加粗/斜体、无序/有序列表、行内代码
  - assistant 消息用 `innerHTML` 输出渲染结果
- 配套 CSS：
  - 标题/strong/code/list 的字号、粗细、间距统一

### 主要文件

- `lab3/js/app.js`
- `lab3/css/style.css`

## Skills 使用

> Skills 不是“为了用而用”，而是用在最省时间、最容易跑偏的节点。

### 1) brainstorming（V0 开始前）

- **适用阶段**：V0（需求拆分、边界、数据结构、API 设计）
- **为什么**：Lab3-2 最容易跑偏成“聊天应用”；brainstorming 能强制把主线拉回“规划系统闭环”。
- **skill模板**：
  - “请把系统拆成模块并给出每个模块最小可用验收标准”
  - “请给出 Plan/Place/Itinerary 的数据表设计与 API 草案”
  - “前端/后端边界怎么划分才不会把逻辑塞进前端？”

### 2) frontend-design（V3/V4/V6 前端页面成型时）

- **适用阶段**：
  - V3：规划表单 + 列表布局（让用户知道下一步）
  - V4：地图选点 + 地点列表（信息层级、按钮位置、空状态提示）
  - V6：行程时间段面板 + AI 总结展示（避免堆 JSON）
- **为什么**：文档强调“状态清楚、流程清楚、结果展示清晰”，这正是 UI/UX 的工作。
- **skill模板**：
  - “请给我一个规划页布局草图：左侧规划列表，右侧表单/地图/行程/AI 总结怎么排？”
  - “空数据/加载中/失败提示应该怎么写才不啰嗦但清楚？”

### 3) git-commit（每完成一个版本就用一次）

- **适用阶段**：V1/V2/V4/V5/V6 完成后
- **为什么**：实验要求提交记录清晰；你每版做完就用一次，能自动生成更规范的 commit message（避免最后补提交、提交粒度太大）。
- **skill模板**：
  - “请根据当前 diff 给出本次提交建议：要提交哪些文件、commit message 怎么写更贴合 Vx 阶段？”

  - 基本信息（标题/日期/预算/人数/偏好）

  - `backend/main.py`
  - `backend/exporter.py`
  - `frontend/src/pages/PlanPage.vue`
  - `frontend/src/api/export.js`

### 扩展 3：更丰富的规则检查（预算/安排完整性/雨天风险提醒）

- **目标**：在“提交前”帮助发现明显问题（没填日期、没加地点、没安排时间段、雨天风险等），避免只靠人眼检查。
- **实现**：
  - 后端新增接口：`GET /api/plans/{plan_id}/checks`
  - 返回结构化 `issues` 数组：`{ level, code, title, detail }`
  - 前端在「AI 总结」区域增加「规则检查」按钮，并展示检查结果列表
- **当前规则（启发式）**：
  - 缺少日期 / 没有地点
  - 部分地点缺少 `adcode`（天气不可稳定查询）
  - 地点未安排时间段
  - 单时间段地点过多（过于紧凑）
  - 人均预算偏低（建议减少跨区移动/选更省钱活动）
  - 天气状态包含“雨”→ 提示雨具与室内备选
- **涉及文件（Lab3-2）**：
  - `backend/main.py`
  - `backend/checks.py`
  - `frontend/src/pages/PlanPage.vue`
  - `frontend/src/api/checks.js`

## 对话中“什么时候用哪种 agent”最合适（建议）

> 下面是把你这次从 Lab3-1 到 Lab3-2 的对话流程，映射到最合适的 agent 类型（用对了会明显省时间）。

### 1) explore（只读快速摸底）

- **适合你对话中的阶段**：
  - “现在无法显示地图，为什么？”（快速定位配置/引用/环境变量）
  - “阅读文档后告诉我应该有哪些步骤？”（在仓库内找文档/已有实现）
  - “项目里哪里做了 weather/llm 调用？”（跨文件找入口与调用链）
- **用法建议**：
  - 先让它输出“相关文件清单 + 每个文件负责什么”，再决定改哪里，避免盲改。

### 2) generalPurpose（实现 + 改代码 + 串联）

- **适合你对话中的阶段**：
  - V2/V4/V5/V6 这种“要落地一个完整功能闭环”的版本迭代
  - “实现带天气上下文的 AI 对话”“后端统一天气服务”“行程保存与展示”
- **用法建议**：
  - 你每次只给一个明确验收点（输入/输出/按钮/接口），它会更稳、更快。

### 3) browser-use（需要真实页面交互验证）

- **适合你对话中的阶段**：
  - UI 改完后想确认：按钮位置、滚动、响应式、地图点击交互、空状态文案是否会遮挡
  - “这里的 UI 非常不好看，改一下”之后想要快速验收视觉与交互
- **用法建议**：
  - 让它执行固定脚本：打开页面 → 新建规划 → 选点加入 → 安排时间段 → 导出 → 规则检查 → 生成 AI 总结。

### 4) cursor-guide（只在问 Cursor/Rules/设置时用）

- **适合你对话中的阶段**：
  - “根据文档生成全局 rules 文档规范”
  - “怎么保证 env.local 不被提交”
  - 任何涉及 `.cursor/rules/`、IDE 设置、Cursor 行为的问题
- **用法建议**：
  - 目标是拿到“官方/标准做法”，避免凭经验猜导致来回试错。

### 5) best-of-n-runner（隔离实验/多方案对比）

- **适合你对话中的阶段**：
  - “UI 方案 A/B 哪个更简洁？”、“导出格式 md 模板 2 套想对比”
  - 或者“规则检查策略要不要更严格？”想多版本试验
- **用法建议**：
  - 用隔离分支同时跑 2 套方案，最后挑一套合并，避免把主分支改乱。
