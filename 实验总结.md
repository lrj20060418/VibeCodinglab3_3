## 实验总结

这次实验做的是“地图选点 → 查天气 → 问 AI”。我觉得重点不是某一个 API 难，而是把它们串起来，并且在出错时也能提示错误信息

### 1）API 串联的体会

- **地图（高德 JS API）**：在地图上点一下，拿到经纬度；再用逆地理编码拿到地名和 `adcode`。
- **天气（高德 Web 服务）**：用 `adcode` 调用天气接口，拿到实时天气并展示（状态、温度、风、湿度、时间）。
- **大模型（chat/completions）**：把当前地点和当前天气作为背景信息，回答用户问题（比如穿衣、出行建议）。

我在实现时注意了两点：

- **依赖关系要清楚**：天气查询需要 `adcode`，所以必须先逆地理编码成功；失败就直接提示“无法查询天气”，不要让页面一直转圈。
- **失败要能解释**：接口失败时，不只在控制台报错，也在页面上给出提示（例如“天气获取失败 / 缺少 adcode”），方便排查。

### 2）Prompt的体会

为了让 AI 的回答更贴合当前地点，我在每次提问时都会把“选中的位置 + 当前天气”一起带上去：

- system 里先说明 AI 的角色（生活出行助手），回答要中文、简洁、可操作。
- 再用 system 放一段上下文（地名、经纬度、天气数据）。

这样做之后，AI 一般能更稳定地根据天气给建议，不容易跑题。

### 3）安全问题的体会

这次是纯前端项目，所以 Key 很难做到完全不暴露。我能做到的是：

- **提交时不放真实 Key**：代码里用占位符，README 里写清楚本地怎么填。
- **分开不同 Key**：地图用 JS API Key，天气用 Web 服务 Key，方便管理，也更好控制权限。
- **如果要真正上线**：应该用后端代理把 Key 放到服务端，并做限流等保护
