## 1. 规范化 AI 操作：推荐步骤（从两份 PDF 提炼）

> 目标：用 Trae/AI **规范地**协作开发一个前后端分离的小系统，而不是“让 AI 一口气写完所有代码”。

### 1.1 先明确目标与边界（避免跑偏）

- 你要做的是：**智能出行规划器**  
- 它不是：地图展示页 / 天气查询页 / AI 聊天页  
- 系统核心主线：围绕“一份规划”组织数据、页面与交互闭环（创建→编辑→保存→再打开）

### 1.2 采用“多轮对话迭代”，每轮只做一个子目标

文档建议：**多轮对话开发，每轮聚焦一个子目标**。  
原则：每轮都要能“跑起来 + 验收”，再进入下一轮。

### 1.3 先定前后端职责边界，再写代码

- **前端**：页面展示、地图交互、表单输入、状态切换、结果呈现  
- **后端**：数据管理、业务逻辑、外部 API（天气/大模型等）统一调用、规划保存与读取  
- 核心要求：**前端不应直接暴露大模型 API Key**，尽量由后端统一处理第三方调用（BFF 思路）

### 1.4 按推荐顺序迭代：先可用，再完善

文档推荐开发顺序（强烈建议照做）：

1. 初始化前后端工程结构（推荐栈 Vue3 + FastAPI + SQLite）
2. 先做后端最基本数据读写（规划保存/读取）
3. 再做地图选点与前端展示（地点管理）
4. 接入天气并展示在规划页面
5. 加入时间段安排与规划编辑能力
6. 最后加入 AI 辅助总结（AI 是辅助，不是主角）

### 1.5 每一轮都写清：输入/输出/验收标准/异常处理

文档强调“工程实现质量”包括：

- loading 状态、请求失败、空数据提示要清楚
- 不要把 JSON 原样堆在页面上，要做“对用户友好”的展示
- 前后端边界清晰，不要把全部逻辑塞进前端

### 1.6 主动使用 Skills（至少一次，用对地方）

知识文档推荐：

- **brainstorming**：开发前拆需求/模块/边界/数据结构
- **frontend-design**：页面布局、地点卡片、行程面板、天气展示区域
- **git-commit**：阶段完成后整理提交信息与提交粒度

### 1.7 过程记录与交付同步做（避免最后补材料）

提交材料（文档要求）通常包括：

- 项目代码（前端 + 后端 + 运行说明）
- AI 协作记录（导出的聊天记录；能看出使用了哪些 Skills）
- 版本管理记录（`git log --oneline` 截图）
- 实验总结（前后端职责、数据管理、AI 角色等反思）

## 2. 一步步版本化需求文档（V0 → V6）

> 你可以把下面当成“迭代路线图”。每一版都包含：你要做什么、要给 AI 的提示词、验收标准。

### V0：需求与系统设计（只做文档，不写代码）

#### 你要做什么

- 写清完整流程：创建/打开规划 → 填基本信息 → 地图选点加入规划 → 展示天气 → 时间段安排 → 保存与再次打开 → AI 辅助总结
- 定前后端职责边界
- 定数据结构草案（规划/地点/行程）
- 定 API 草案（后端路由、入参出参）

#### 给 AI 的提示词（可复制）

> 你是我的系统设计助手。请根据 Lab3-2 要求，输出：  
> 1）模块划分（规划管理/地点管理/天气展示/行程安排/AI辅助总结）  
> 2）前后端职责边界  
> 3）SQLite 表设计草案（Plan/Place/Itinerary…）  
> 4）REST API 设计（路由 + 入参 + 返回示例）  
> 5）每个模块的最小验收标准（可检查项）。

#### 验收标准

- 有一份能指导开发的设计文档（模块/数据/API/验收明确）

---

### V1：工程初始化 + 目录结构（前后端能启动）

#### 你要做什么

- 建立 `frontend/` 与 `backend/`
- 前端能启动并打开页面
- 后端能启动并提供 `/health`

#### 给 AI 的提示词

> 请按推荐栈初始化前后端分离工程：  
> - frontend：Vue3，可启动  
> - backend：FastAPI，提供 GET /health 返回 ok  
> 给出运行方式与端口约定。

#### 验收标准

- 前端能打开页面
- 后端访问 `/health` 返回 ok

---

### V2：后端规划管理最小闭环（先把数据“存起来”）

#### 你要做什么

- 后端实现规划 CRUD（至少：创建/列表/详情/更新）
- SQLite 持久化（不能只存内存）
- 返回结构化 JSON，错误返回清楚

#### 给 AI 的提示词

> 请优先实现后端规划管理：  
> - SQLite 持久化  
> - API：创建规划、获取详情、更新、列表  
> - 参数校验与错误处理（不存在/缺字段）  
> - 返回给前端友好的 JSON。

#### 验收标准

- 重启后端后，规划数据仍能读取（持久化成立）
- CRUD 接口可用

---

### V3：前端规划表单 + 打开/保存（用户知道下一步）

#### 你要做什么

- 前端提供“新建/编辑规划”的表单（日期、预算、人数、偏好等）
- 与后端规划接口联调
- 有 loading/失败/空状态

#### 给 AI 的提示词

> 请实现前端规划页面：  
> - 表单字段：日期/预算/人数/偏好  
> - 支持新建、保存、再次打开编辑  
> - 状态清楚：loading/失败/空数据  
> - 用户流程清晰：下一步提示明显。

#### 验收标准

- 用户可创建 → 保存 → 刷新 → 再打开继续编辑

---

### V4：地点管理（地图选点加入规划、可查看/删除）

#### 你要做什么

- 地图选点生成地点对象（名称/经纬度/adcode…）
- “加入规划”后保存到后端（规划数据的一部分）
- 支持查看/删除（最小可用）

#### 给 AI 的提示词

> 请实现地点管理模块：  
> - 前端地图选点，生成地点对象  
> - “加入规划”调用后端保存并持久化  
> - 地点列表可查看与删除  
> - 有 loading/失败/空状态。

#### 验收标准

- 地点是真正存进规划里，刷新/重启后仍存在

---

### V5：天气展示（为规划服务，不是独立查询页）

#### 你要做什么

- **后端统一调用**天气 API（不要让前端到处拼第三方请求）
- 前端展示与规划地点相关的天气信息（或摘要）
- 处理失败/空状态

#### 给 AI 的提示词

> 请把天气调用放到后端统一处理：  
> - 输入：地点（adcode 或经纬度，按你设计）  
> - 输出：前端友好的天气摘要字段  
> - 前端在规划页面展示天气（至少能看到当前规划地点的天气）  
> - 处理 loading/失败/空数据。

#### 验收标准

- 规划中的地点能看到天气；失败时提示原因

---

### V6：行程安排 + AI 辅助总结（AI 是辅助，不是主角）

#### 你要做什么

- 行程按时间段组织（上午/下午/晚上）
- AI 辅助总结：基于地点/天气/时间安排/预算生成建议
- **大模型 Key 只在后端**（前端不能暴露）

#### 给 AI 的提示词

> 请实现“AI 辅助总结”模块：  
> - 后端调用大模型（Key 放后端配置/环境变量，不暴露给前端）  
> - 输入：规划的地点、天气、时间段安排、预算等结构化信息  
> - 输出：一段自然语言建议（优点/风险/可改进点）  
> - 前端展示总结，并有 loading 状态  
> - 给出清晰 prompt（system + context），保证不跑题。

#### 验收标准

- AI 总结能稳定基于当前规划输出（不是泛泛而谈）
- 前端代码中找不到大模型 Key

## 3. 每轮统一的“规范对话模板”（强烈推荐）

你每次和 AI（Trae）对话，建议按这个结构发：

1. **本轮目标**：一句话（例如“实现后端规划 CRUD 并用 SQLite 持久化”）
2. **范围**：做什么 / 不做什么（例如“不做 UI 美化，不做拓展功能”）
3. **接口与数据**：需要哪些路由、入参出参、数据字段
4. **异常与状态**：loading / 失败 / 空数据怎么表现
5. **验收标准**：3–5 条可检查项
6. **约束**：前后端边界、不能暴露 Key、尽量后端统一调用第三方 API

你可以直接复制下面这一段当模板：

```text
本轮目标：
范围（做 / 不做）：
接口与数据（路由 / 入参 / 返回）：
异常与状态（loading / error / empty）：
验收标准（3-5 条）：
约束（前后端边界 / 不暴露 key / 其它）：
```

## 4. 你接下来该怎么做（最简行动清单）

如果你准备开始 Lab 3-2，建议你按顺序做：

1. 先用 V0 产出“模块/数据/API/验收”文档（10–20 分钟）
2. 再做 V1 工程初始化（能启动）
3. 先做 V2（后端 CRUD + SQLite）保证“规划能保存并再打开”
4. 再逐步接入地图（V4）、天气（V5）、时间段（V6 的前半部分）
5. 最后再做 AI 辅助总结（V6 的后半部分）
6. 每完成一版就提交一次 Git，并把对话记录存档（避免最后补材料）
